# Stellara 문서 시작 가이드

이 폴더는 **프로젝트를 이해하고, SDD를 이어서 쓰고, 실제 작업을 집어가는 기준 문서 묶음**입니다.

비개발자, 개발자, AI 협업자 모두 아래 순서로 보면 가장 빠르게 맥락을 잡을 수 있습니다.

## 문서 역할

### `MVP.md`

- 이번 버전에서 반드시 할 것 / 미뤄도 되는 것
- 기능 우선순위
- 주차별 큰 흐름

이 문서는 **범위 문서**입니다.

### `SDD.md`

- 현재 코드 기준 구조
- API / DB 설계
- 리스크
- 실행 우선순위
- 일정

이 문서는 **기술 설계 문서**입니다.

### `TASKS.md`

- 협업자가 바로 가져갈 수 있는 작업 단위
- 선행 작업
- 예상 수정 파일
- 담당 역할

이 문서는 **실행 문서**입니다.

### `NOTION_EXPORT.md`

- 다른 사람에게 공유하기 쉬운 통합 요약본
- 노션에 한 번에 붙여넣기 좋은 버전

이 문서는 **공유용 문서**입니다.

### `SDD_input_codebase_overview.md`

- 코드베이스를 읽고 정리한 분석 메모
- SDD 초안의 근거 자료

이 문서는 **참고 자료**입니다.

## 누가 무엇부터 보면 되나

### 개발자 / AI 협업자

1. `README.md`

추천 목적:

- 현재 코드 상태와 문서 상태를 같은 기준으로 맞춤
- 선행 작업이 충족된 이슈부터 바로 착수

## SDD를 어떻게 시작하면 되나

SDD 작업은 아래 순서로 시작하면 됩니다.

1. **범위 확인**
   `MVP.md`에서 이번 주기에서 반드시 포함할 기능과 제외할 기능을 확인합니다.

2. **현재 상태 확인**
   `SDD.md`의 현재 구현 상태, API 설계, DB 설계, 리스크를 확인합니다.

3. **작업 단위 선택**
   `TASKS.md`에서 선행 작업이 없는 이슈 또는 이미 풀린 이슈의 다음 작업을 고릅니다.

4. **작업 중 문서 갱신**
   작업하면서 결정이 생기면 문서를 같이 업데이트합니다.

## 무엇이 바뀌면 어떤 문서를 고쳐야 하나

### 기능 범위가 바뀔 때

- `MVP.md`

예:

- 이번 버전에서 공유 기능을 빼기로 함
- 카카오 로그인을 후순위로 미룸

### API / DB / 구조가 바뀔 때

- `SDD.md`

예:

- 출생지 geocoding 방식 변경
- `friendCodes` 컬렉션 구조 변경
- `chartVersion` 규칙 변경

### 작업 쪼개기 / 담당 분배가 바뀔 때

- `TASKS.md`

예:

- 한 작업이 너무 커서 둘로 나눔
- FE / BE 역할 분리가 달라짐

### 실행 방법 / 온보딩 방법이 바뀔 때

- 루트 `README.md`
- 필요하면 `docs/README.md`

예:

- fixture 기본값 변경
- Prokerala backup key 운영 방식 변경
- Firebase 실행 준비 방식 변경
- Flutter 실행 명령 변경

## 추천 작업 시작 순서

현재 기준으로는 아래 순서가 가장 자연스럽습니다.

1. `TASKS.md`의 `T01` ~ `T04`
   Prokerala 실응답 검증과 parser 안정화

2. `T05` ~ `T15`
   Firebase / 출생정보 저장 / 좌표 변환 / 차트·운세 캐시

3. `T16` ~ `T21`
   친구, 즐겨찾기, 궁합 실데이터 연결

4. `T22` ~ `T24`
   AI 질문 + 후반부 디자인/공유 선행 준비

5. `T25` 이후
   공유, 디자인 마감, 마이페이지 수정, 발표 준비

## 현재 임시 운영 규칙

- 기본 개발 모드는 `USE_FIXTURE_IN_DEBUG=true`
- Prokerala 실응답 검증이 필요할 때만 실 credential 을 사용
- Prokerala credential 은 `primary → seoyeon → seonwoo → doyeon` 순서로 backup fallback 가능
- 이 backup credential 운영은 무료 플랜과 짧은 프로젝트 기간을 고려한 **임시 운영 규칙**이다
- MVP 기본 경로는 `Spark-only` 이며 `Cloud Functions` 는 포함하지 않는다
- AI 질문은 메인 화면에서 `자동(추천) / GPT / Claude` 수준만 노출하고, 실제 model id 는 `.env` 에서 관리한다
- AI provider 기본 흐름은 `OpenAI → Anthropic → local fallback` 순서를 권장한다
- `데모 데이터로 둘러보기` 버튼은 최종 마감 직전까지 유지한다
- 출생지 좌표 자동 변환은 현재 Android/iOS에서 우선 지원하고, Web은 보조 타깃으로 본다

## 후반부 화려한 기능을 어떻게 다루는가

현재 문서 기준 원칙은 아래와 같습니다.

- 공유, 애니메이션, 컬러 디자인은 **후반부 구현 대상**
- 하지만 13주차에 몰리지 않도록 **12주차 후반부터 와이어프레임, 디자인 토큰, 자산 검토를 먼저 시작**

즉, “후반부 구현”과 “막판 과밀 방지”를 동시에 만족하는 방식으로 운영합니다.

## 문서 사용 원칙

- 문서는 코드보다 늦게 업데이트하지 않습니다.
- 비개발자도 읽을 수 있는 표현을 우선합니다.
- 기술 세부사항은 `SDD.md`에, 실행 단위는 `TASKS.md`에 둡니다.
- 문서가 충돌하면 먼저 충돌을 정리한 뒤 작업합니다.
